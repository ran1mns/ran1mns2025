Titre du Cours : Introduction au Versioning avec Git et GitHub

Introduction

Git est un système de contrôle de version distribué, un outil indispensable pour suivre les modifications de votre code. GitHub est une plateforme web qui héberge des dépôts Git et fournit des outils pour faciliter la collaboration entre développeurs.

1. Configuration Initiale

Avant toute chose, vous devez configurer Git avec vos informations.


# Configurer votre nom d'utilisateur et votre email
git config --global user.name "Votre Nom"
git config --global user.email "votre.email@example.com"

Pour commencer un projet avec Git, vous avez deux options :

  • git init : Crée un nouveau dépôt Git dans un dossier existant.
  • git clone [URL] : Copie un dépôt distant (par exemple depuis GitHub) sur votre machine locale.

2. Le Cycle de Travail de Base

Le travail local avec Git suit un cycle en trois étapes :

  1. Modification : Vous modifiez vos fichiers dans votre répertoire de travail.
  2. Indexation (Staging) : Vous sélectionnez les modifications que vous voulez inclure dans le prochain "snapshot" (commit) avec git add.
  3. Commit : Vous enregistrez ce "snapshot" de manière permanente dans votre historique avec git commit.

# Voir le statut des modifications
git status

# Ajouter un fichier à l'index
git add nom_du_fichier.html

# Ajouter tous les fichiers modifiés
git add .

# Enregistrer les modifications avec un message descriptif
git commit -m "feat: Ajout de la page d'accueil"

3. Travailler avec les Branches

Une branche est une ligne de développement indépendante. C'est une pratique essentielle pour développer de nouvelles fonctionnalités sans affecter la version stable du code (qui se trouve généralement sur la branche `main` ou `master`).


# Lister les branches existantes
git branch

# Créer une nouvelle branche et s'y placer
git checkout -b ma-nouvelle-fonctionnalite

# Revenir sur la branche principale
git checkout main

Fusionner les Branches

Une fois votre fonctionnalité terminée, vous devez la rapatrier sur la branche principale. Il y a deux manières principales de le faire :

  • git merge : Crée un nouveau "commit de fusion" qui combine l'historique des deux branches. C'est simple et préserve l'historique exact.
  • git rebase : Réapplique les commits de votre branche au sommet de la branche principale. Cela crée un historique linéaire plus propre, mais modifie l'historique, ce qui peut être risqué si la branche est partagée.

4. Collaboration avec GitHub

GitHub facilite le travail en équipe sur un même projet.

  • git push : Envoie vos commits locaux vers le dépôt distant sur GitHub.
  • git pull : Récupère les dernières modifications du dépôt distant et les fusionne avec votre branche locale.
  • Pull Request (PR) : C'est le cœur de la collaboration sur GitHub. C'est une demande pour intégrer vos modifications (votre branche) dans une autre branche (généralement `main`). Cela permet aux autres membres de l'équipe de relire votre code (code review) avant la fusion.

Exercice Pratique : Workflow de Collaboration

  1. Créez un dépôt sur GitHub.
  2. Clonez ce dépôt sur votre machine locale.
  3. Créez une nouvelle branche nommée ajout-readme.
  4. Sur cette branche, modifiez le fichier README.md.
  5. Faites un commit de vos modifications.
  6. Poussez votre branche sur GitHub.
  7. Depuis l'interface de GitHub, ouvrez une Pull Request pour fusionner votre branche dans `main`.
  8. Fusionnez la Pull Request.
  9. En local, revenez sur la branche `main` et faites un pull pour récupérer les modifications.
